2. 技术路线讨论


2.1 架构范式：AR + DiT




提问指出 Genie 3 偏纯自回归，业内通用做法是 AR + DiT（有逐帧 DiT、有逐块 DiT）。受访者表示**"AR + DiT 这套范式应该是比较公认的了"，但不清楚 X Center（王恒团队）具体采用逐帧还是逐 block；IEG 侧方案未定，但个人偏向逐帧**。




2.2 逐帧 vs 逐块（block/chunk）




延迟：逐块方案每次动作需推理多帧，延迟会被帧数"卡死"；逐帧方案的延迟主要来自推理效率本身，因而逐帧延迟更低。


反方观点：若逐块一次性算得更多、吞吐更快，可摊薄延迟——受访者认为这是"比较伪的"说法；在同等计算量前提下逐块必然更慢。


画质：受访者认为"逐块展开 OK 而逐帧不 OK，说明长时序能力不是真长时序"，终局应做逐帧展开；第一阶段逐帧保画质没有问题。




2.3 画质目标




不对标可灵、Seedance 的画质——"目标没那么高，也不可能达到"，因为对方是双向（非因果）一次性生成，天然更好。


落地 baseline：达到并超过手游画质（若要在游戏内落地，至少要比手游更强）。




2.4 蒸馏范式的争议




批评现状：以当前最火的 Long Live 为代表——teacher 不具备长视频展开能力，却直接蒸馏出 student，受访者认为不合理；"教师没有的能力想让学生凭空拥有，不太靠谱"。


昆仑万维案例：已刷到 720P / 40 FPS，但质量不好；其加速效果以牺牲模型规模等为代价，且"整个训练范式可能都不太对"，至少从已发表学术工作看蒸馏范式尚未完备。


受访者倾向的路线：先构建原生视频流模型（一开始就具备流式生成能力的教师），再做蒸馏。




2.5 延迟—画质—成本权衡




三者确实难以兼得；团队当前优先级：可控性 > 画质 > 速度。


加速手段（预训练阶段引入加速、强剪枝、量化、蒸馏等策略）在 AR 实时推流模型中尚未被充分应用，预计后续会有大量工作把加速"研究透彻"。

当前主流追求"理解与生成一体化"（一个模型兼具逻辑与感知），受访者认为理论上世界模型确实希望如此，但游戏落地必须解耦。

进一步观点：视频本质是持续感知，physics 只是碰巧被持续感知满足；把不该由视频承担的东西（物理、逻辑）强塞进视频模型，"只是没有更好方案时的妥协，不是长久之计"。


受访者自述"做技术比较保守"，认为 Genie 这类是"玩具"；该观点为个人立场，团队内并非人人前瞻一致，但有相当部分同事认同。






5. 商业化与应用场景


5.1 为什么不能与游戏渲染管线正面竞争




传统游戏渲染管线（上世纪八九十年代提出）极其成熟，渲染成本极低；游戏内容生产贵，恰恰是因为要用极低成本做出 3D 画质，导致整个 workflow 笨重。


若 AI 模型只是"效果与游戏一样好、成本差不多"，即使动态性更好，也毫无竞争力——同质化竞争没有前途。




5.2 正确的方向：To C + Agent 化




世界模型应面向 To C：与 code agent 等能力结合，做成"普通小白就能生产、就能玩"的产品。


受访者个人动机：给孩子讲故事时，希望丢进一个故事（如"青蛙王子"）就能生成可进入游玩的交互式视频场景——这种需求若走传统游戏管线极其笨重，AI 方案即使贵也值得。


澄清：这类场景中模型提供的是可交互的世界/空间，玩法机制等属于产品层问题，不由模型单独解决——这也呼应其"解耦"主张。




5.3 概念辨析




两个团队对外都叫"世界模型"，实质都是互动视频生成；受访者认为"互动视频生成"叫法更精确，但行业已统称世界模型，遂从俗。






6. 3D 生成领域（受访者本行）


6.1 "3D 动不起来"是根本死结




两种"动法"都行不通：



生成 3D 资产 + 骨骼绑定/驱动动画：要做到鲁棒"不可能"——没有数据，且绑定、蒙皮、动画完全是人工作业产物，不是可端到端学习的天然信号；上限只能做到"静态资产生成可看"。


李飞飞 Marble 等高斯/自回归高斯路线：本质是重建，而只要重建就有大误差，上限被锁死。






受访者自述观点可能偏主观（"不一定对"），但供参考。




6.2 3D 厂商对比


厂商/产品	评价
VAST / Tripo（H3.1、P1 方案）	最近做得不错；P1 方案轻巧快、纹理质量够 UGC 用；泛化性、结构优于混元；UGC 赛道领先；综合质量与混元打平，同处第一梯队
混元 3D	整体质量不错；内部未开放的 3.5 模型质感预计优于 Tripo；规整度、细节质感更强
Meshy（转写"麦是/梅西"）	"吹得不错"，实际全方位不讲究：边缘不锐、结构不规整（圆变椭圆）、重复 pattern（如铠甲）断裂不重复、结构不明确；小白看着差不多，游戏从业一眼看出不行；对生产管线远远不够，UGC 随便摆可以
Hyper3D（音"海特麦"）、Rodin（影眸）	排第一梯队之后；但 UGC 场景不敏感，差距不影响使用


6.3 3D 生成的三大类需求




UGC（如 Roblox）：即生成即可用；最大瓶颈不在 3D AIGC 本身，而在资产能否压缩成低成本包、游戏引擎 infra 是否支持。


白盒验证（PGC 中间环节）：正式资产制作前搭建场景白盒，验证可玩性（如 CS 类地图的路口宽窄、视野）；AI 资产用于向策划演示与摆入白盒验证。


游戏生产管线（唯一有明确商业价值的方向）：当前几何、纹理、拓扑模型均达不到美术要求，只能作参考，加速价值有限；各家刚起跑，尚无任何方案能直接产出进入 3A 管线的最终资产（最多做原画/过渡工具）——但一旦打通，商业前景最大。




6.4 混元文生视频（V6）与数据问题




混元 V6（对标可灵、Seedance 的双向扩散文生视频）2024 年已发布，目前在做镜头/分镜能力（跟随 Seedance）；受访者判断落后较多。


怀疑数据有问题："从一开始到现在都不太对"；旁证：混元 Image（与视频同团队）指令遵循能力强，但画质未必优于千问（音"纤维"）等。


实时互动视频的数据观：



做游戏方向不需要太多真实数据，游戏内采集即可；


做通用方向（如灵波瞄准具身智能）则需大量真实数据；


若基座足够好，中训练更像 warmup——别把基座能力训丢；


长视频展开是能力缺口（短 clip 具备的能力，长视频未必有）；


未来数据会成为瓶颈：长时互动的"生动感"与数据强相关。

4.2 核心技术概念


术语	讲解
世界模型 vs 互动视频生成	业界统称"世界模型"（World Model），指可交互、可持续推演的视频生成系统；受访者认为"互动视频生成"是更准确的叫法——本质是接收动作/文本信号后实时逐帧生成画面的模型。
AR（Autoregressive，自回归）	按时间顺序逐帧（或逐块）预测下一帧的生成方式，天然适配流式/实时场景；代价是误差累积导致长视频易崩坏。
DiT（Diffusion Transformer）	用 Transformer 做扩散模型去噪网络的架构，是当前视频生成主流骨架（Sora 之后成为标配）。
AR + DiT	业内公认的实时视频范式：自回归负责时间维度的流式推进，DiT 负责每帧（或每块）的画质生成。变体包括逐帧 DiT（延迟低）与逐块 DiT（一次生成多帧，吞吐高但延迟被帧数卡死）。
双向扩散 vs 因果扩散	可灵、Seedance 等文生视频为双向（非因果）一次性生成整段视频，画质上限更高但无法实时交互；实时互动模型必须用因果（自回归）结构，画质天然吃亏。
预训练 / mid-train（中训练）/ 后训练	预训练指从零训练基座视频模型（如 Seedance）；mid-train 指在已有基座上做领域适配与能力注入（如游戏动漫化、action control），是 WorldPlay 1.5 及 X Center 采用的方式；灵波系称"中训"。
Rollout（长视频展开）	模型自回归地持续生成超长视频序列的过程。核心难点：误差累积导致画面崩坏、3D 一致性丢失。
Context Forcing	长视频展开的一种训练策略（WorldPlay 1.5 采用），通过条件化历史上下文维持长时一致性；Rolling Forcing 是其变体（受访者评价不高）。
蒸馏（Distillation）/ Teacher–Student	把大模型（教师）的能力压缩到小模型（学生）以满足实时性。争议焦点：教师本身不具备长视频 rollout 能力却直接蒸馏学生（如 Long Live）被认为不合理；新观点主张师生架构与能力一致，或先构建原生视频流教师再蒸馏。
Long Live	当前热门的实时长视频蒸馏框架（英伟达等参与），受访者以其为"蒸馏范式不完备"的典型。
Omni Control / Action Control	多模态控制信号：键盘/手柄动作、文本、音频等，用于驱动互动视频；X Center 计划扩展更全的 omni control。
Human-in-the-loop	人在回路：游戏美术/策划需在生成过程中精确干预（指定某人某时刻出现在某地），是通用世界模型难以满足、IEG 路线强调可控性的原因。
量化 / 剪枝 / 蒸馏加速	模型推理加速三件套：降低数值精度（量化）、删除冗余权重（剪枝）、小模型学习大模型（蒸馏）。受访者认为这些手段在 AR 实时推流模型中尚未被充分应用。
首帧延迟 / FPS / 720P	实时性的核心指标：首帧响应时间、每秒生成帧数（如昆仑万维 40 FPS）、分辨率（720P）。
Code Agent	编程智能体。受访者设想世界模型与 code agent 结合，让小白用户生成可玩的互动内容（To C 方向）。
具身智能（Embodied AI）	让 AI 在物理/仿真环境中感知与行动。灵波等团队用真实数据训练世界模型以服务具身智能。


4.3 3D 相关概念


术语	讲解
3D 高斯泼溅（Gaussian Splatting）	用大量 3D 高斯椭球表示场景的渲染技术，重建快、可实时渲染，但本质是重建而非生成——存在难以消除的伪影与误差（受访者认为这是其上限锁死的原因）。
重建 vs 生成	重建依赖观测数据还原场景（误差决定上限）；生成（end-to-end）直接从模型产出内容，无启发式中间环节。受访者坚定站生成路线。
骨骼绑定 / 蒙皮动画	让 3D 模型"动起来"的传统工艺：为模型绑骨骼、刷蒙皮权重、制作动画。纯人工流程、无大规模数据，被视为 3D AIGC 最难自动化的环节。
UGC / PGC	用户生成内容 vs 专业生成内容。3D AIGC 当前主要在 UGC（如 Roblox 生态）与 PGC 的白盒验证环节可用。
白盒验证	游戏关卡设计早期用简陋几何体搭建场景验证可玩性（动线、视野、掩体）的环节，AI 生成资产可用于快速填充白盒。
游戏渲染管线	从几何、纹理、拓扑到实时光栅化的成熟工业流程，成本极低、优化极致——是 AI 视频生成进入游戏时面对的"成本地板"。
图生视频打榜	在公开榜单（如 Artificial Analysis）上以图生视频能力排名，是视频模型厂商证明实力的常规手段（Video Rebirth 曾靠前）。
