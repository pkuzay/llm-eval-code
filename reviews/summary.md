# 大模型横向评测真实点评

> 评测口径：本文件不是泛泛模型画像，而是基于仓库内原始输出做的真实验收点评。前端题已通过本地 HTTP 服务在浏览器打开；Python 题已执行 `py_compile` 和 Hunyuan 自带单测；研究题与图片题按事实可靠性、是否读懂题目、表达质量逐项判断。模型原始输出保持不改。

## 验收结论

这一轮最有价值的发现是：模型能力和交付质量不是一回事。有的模型回答思路不错，但保存成文件后不可运行；有的页面视觉包装很强，但加载很慢；有的研究报告读起来非常完整，但关键事实很可能是编出来的。

- **GLM 5.2**：前端视觉和产品包装最强，特别适合做演示型 demo；但研究题里自信给出很多未经验证的事实，代码题也把解释文字混进 `.py` 文件，导致原始交付不可运行。
- **Qwen 3.7 Plus**：结构化表达强，代码块本身通常更标准，产品原型信息密度高；但同样存在交付格式问题，且图片题和研究题容易“先假设再长推理”。
- **Hunyuan 3**：整体最保守，事实性调研里会主动承认不确定性；但前端打磨偏弱，队列题虽然语法可运行，自带并发单测实际失败，多模态表现也不稳定。

投资视角下，我会把它们粗略归类为：**GLM 是产品 demo 型，Qwen 是结构化分析型，Hunyuan 是谨慎初稿型**。但三者都不能直接作为无需验收的生产输出。

## 逐题点评

### 1. 3D 数据可视化地球仪

验收动作：浏览器打开三份 `index.html`，检查首屏文本、canvas、控制项和 console。三份页面都能渲染 canvas，未看到 console error。

| 模型 | 真实点评 |
| --- | --- |
| GLM 5.2 | 最像一个可展示产品，首屏有 `EARTHVIZ GLOBAL DATA OBSERVATORY`、指标按钮、图层开关、视觉参数、年份轴等，DOM 复杂度明显最高。优点是产品化和视觉包装强；风险是代码体量大，后续维护成本高。 |
| Qwen 3.7 Plus | 页面能打开，具备 GDP/人口/航班量、时间轴和城市信息面板；但首屏还残留“正在加载全球数据...”，完成感弱于 GLM。整体是合格实现，但演示冲击力不够。 |
| Hunyuan 3 | 页面简洁，有 canvas、指标切换、时间轴和自动播放按钮，基础需求覆盖；但信息层次和产品感较弱，更像 MVP。 |

结论：**GLM 明显最佳，Qwen 合格，Hunyuan 基础可用。**

### 2. 线程安全泛型有界阻塞队列

验收动作：执行 `python3 -m py_compile` 编译三份 `queue.py`，并运行 Hunyuan 文件内自带单测。

| 模型 | 真实点评 |
| --- | --- |
| Hunyuan 3 | 文件是合法 Python，语法编译通过；队列核心实现使用两个 `Condition`，方向正确。但运行自带单测失败，多生产者测试中 `max_len` 在线程闭包里没有 `nonlocal`，触发 `UnboundLocalError`，最终只消费到 `4/100` 个元素。核心类可能可用，但测试代码有实质 bug。 |
| GLM 5.2 | 原始 `.py` 文件第一行就是中文解释，后面才是 fenced code block，`py_compile` 直接 `SyntaxError`。代码块内部设计较工程化，支持 timeout 和 `wait_for`，但作为仓库交付物不可直接运行，真实验收应扣重分。 |
| Qwen 3.7 Plus | 同样把解释文字和 Markdown 代码块直接写进 `.py`，`py_compile` 直接 `SyntaxError`。代码块内实现比较标准，测试覆盖也合理，但交付格式不合格。 |

结论：**按“原始文件能否运行”看，三者都不合格；Hunyuan 至少能编译但测试失败，GLM/Qwen 需要人工抽代码块后才能评估实现。**

### 3. iOS 18 天气卡片

验收动作：浏览器打开三份页面，检查可见内容、天气切换控件和 console。

| 模型 | 真实点评 |
| --- | --- |
| GLM 5.2 | 首屏视觉最丰富，有晴天/大风/暴雨/暴雪四个按钮，材质、粒子和卡片氛围最接近高级 demo。console 有 Tailwind CDN 生产环境 warning，但不影响演示。 |
| Qwen 3.7 Plus | 页面能打开，也有四种天气按钮，信息组织完整，上海本地化文案更自然；视觉效果比 Hunyuan 丰富，但整体精致度略弱于 GLM。 |
| Hunyuan 3 | 页面静态、干净，呈现 Cupertino 天气卡片，但浏览器中没有明显天气切换按钮，交互完成度不足。适合做静态 UI，不适合拿去展示“多状态天气”。 |

结论：**GLM 最好，Qwen 次之，Hunyuan 是静态卡片级别。**

结论：**若看产品叙事 GLM 最强；若看小程序可用感 Qwen 最稳；Hunyuan 是轻量 MVP。**

### 5. 模型 API 价格调研

验收动作：阅读三份 Markdown 报告，重点看是否承认版本不确定、是否给出可验证来源、是否把假设包装成事实。

| 模型 | 真实点评 |
| --- | --- |
| Hunyuan 3 | 最符合投研工作习惯。开头明确说明 GPT 5.5、DeepSeek v4、Claude Opus 4.7、Kimi 2.6、HY3 preview、Gemini 3.1 pro 这些版本号多数没有公开正式信息，然后用现役模型对标。这种“不知道就说不知道”的姿态更可信。 |
| GLM 5.2 | 给出非常完整的价格表和选型建议，但 GPT 5.5、Claude Opus 4.7、Gemini 3.1 Pro 等具体版本和价格缺少可验证来源。读起来专业，事实风险很高。 |
| Qwen 3.7 Plus | 同样直接给出 2026 最新价格，如 GPT-5.5 1M 上下文、DeepSeek V4 Flash/Pro、HY3 preview 价格等，很多信息需要官方核验。优点是表格清晰，缺点是“精确幻觉”风险最大。 |

结论：**事实性调研 Hunyuan 胜；GLM/Qwen 只能作为线索草稿，不能直接用于投资判断。**

## 模型横向判断

### GLM 5.2

强项是前端 UI、产品包装、演示型页面。它愿意给大量视觉细节和功能模块，天气卡、地球仪、食堂约饭都很有展示感。

主要风险是可靠性。它在研究题里会把未经验证的信息写得非常完整，在代码题里也没有把最终交付整理成可运行 `.py`。它适合做“第一版好看 demo”，不适合直接做事实依据。

### Qwen 3.7 Plus

强项是结构化表达、信息组织和较稳的代码思路。食堂约饭页面很像真实小程序，研究题给的线索也多。

主要风险是过度推理。图片题里一旦基础识别错了，仍会继续给很长的逻辑链；研究题里也容易给大量精确但未验证的信息。它适合做“信息扩展器”，但必须配事实核查。

### Hunyuan 3

强项是谨慎和边界感。价格题里主动承认版本不可查，这是三者里最像投研助手的表现。

主要风险是产品打磨和执行细节。前端页面偏 MVP，队列题虽然语法可运行但自带单测失败，多模态题时好时坏。它适合做“低风险初稿”，但不适合承担高展示要求或强多模态任务。

## 最终投资视角结论

1. **做产品 demo / 前端原型**：优先看 GLM。它在视觉包装和完整度上最突出，但要接受代码体量大和加载性能不稳定。
2. **做结构化分析 / 信息扩展**：优先看 Qwen。它能给出很多线索和完整框架，但要建立事实核查流程。
3. **做事实性调研初稿**：优先看 Hunyuan。它更愿意承认不确定性，适合作为保守底稿。
4. **做代码交付**：三者都不能只看回答内容，必须跑编译和测试。当前队列题原始交付里，GLM/Qwen 甚至不是合法 Python 文件，Hunyuan 测试失败。
5. **做多模态/视觉推理**：必须人工验收。GLM 有时诚实拒答、有时能推理；Qwen 容易套模板；Hunyuan 有时会给具体识别但真假难辨。

下一步建议给每道题增加一张“验收卡”：运行结果、截图、事实核验状态、最终人工评分。这样这个仓库会从“模型输出收集”升级为真正可复用的投资评测资产。
